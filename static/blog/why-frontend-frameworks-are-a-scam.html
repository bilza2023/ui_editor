🧨 Why Modern Frontend Frameworks Are a Scam

13 July 2025

Everyone says: "Use React, use Svelte, use Vue — they're the future!" But the deeper you go, the more you realize...

🔍 The Core Problems

1. You end up babysitting your state

Every developer eventually writes a function like redraw() that exists solely to poke the framework and force a UI update. Why? Because setting a nested property, modifying an array, or tweaking an object silently fails to trigger reactivity. You’re not managing state — you’re tricking the framework into thinking you did. It becomes a cycle of cloning, spreading, and reassigning — all to keep the illusion alive.

Real reactivity should be automatic and consistent. If you're spending more time making the framework notice your changes than writing actual logic, then it's not helping — it's babysitting.

2. "Surgical DOM updates" — for what?

Frameworks boast that they surgically update the DOM using diffing algorithms, fiber trees, and shadow nodes. But most apps — especially educational ones — just show a few headers, paragraphs, and images. Why do we need a surgeon to rearrange a shopping list? The complexity is unnecessary. A simple innerHTML or template refresh would suffice.

Optimizing for performance only matters when you're hitting performance bottlenecks. For most use cases, the DOM isn’t the problem — it’s the overengineering that is.

3. Web components? It’s called a webpage.

Framework evangelists praise web components for bundling HTML, JS, and CSS together. But the web already had that — it’s called a .html file. A webpage is the original component: self-contained, renderable, cacheable, and portable. No build step needed. No runtime needed. Just open and it works.

You don’t need a shadow DOM to isolate styles. You need clarity and discipline — something a well-structured webpage offers natively.

4. Fragmented views, not cohesive apps

In frameworks like Svelte and React, the "app" is just a bundle of disconnected components. Each one is reactive in isolation, but there's no unifying backbone. You can’t easily reason about global state, cross-component flow, or app-level structure.

You end up with dozens of mini-apps — each .svelte or .jsx file doing its own thing. The result isn’t an application. It’s a patchwork. A network of floating UI islands with no shared architecture.

5. Reactivity ≠ reactive

Frameworks claim to be reactive, but they fail on the most basic interactions. Set a property in an object? Doesn’t work. Update a specific index in an array? Nothing happens. Unless you know the exact magic dance — clone, spread, reassign — the view won't update.

This isn’t reactivity. It’s a guessing game. You become a state mechanic, patching and poking your data instead of working with it directly.

6. One reactive statement = total rehydration trap

You use a simple reactive expression like {$count} and suddenly you're thrown into the world of SSR, hydration mismatches, lifecycle bugs, and race conditions. What started as a small template grows into a minefield of edge cases between server and client.

Why are you even in this debate? You wanted a live counter. You got a hydration debugging session. That’s not productivity — it’s overhead in disguise.

7. In React/Svelte you invent structure. In Express, you inherit it.

Express gives you rails to follow: routers, controllers, services, views. It pushes you toward discipline — whether you like it or not. React and Svelte? They give you nothing. No file structure. No conventions. Just a blank slate and a lot of opinions to choose from.

You spend your first days inventing patterns. Then weeks fixing what those patterns broke. A real framework should reduce uncertainty, not create it.

8. You build an app. Then spend 100 hours fighting the framework to make it feel like an app.

You start with enthusiasm. The first component looks great. But then you want routing, persistent state, modals, loading indicators, and shared logic. That’s when things break. You start gluing libraries together. Hacking the store. Passing props five levels deep.

What you built isn’t an app — it’s a shell. The hard stuff remains unsolved, and the easy stuff was never the problem to begin with.

💡 Final Realization

You wanted a reactive interface. But instead of reactivity, you got fragility. The closer you got to shipping, the more the framework fought back. You stopped building features and started fixing abstractions.

Reactivity was supposed to help you — not become your job title.

🧭 The Return to Earth

  • Express + EJS → gives you real structure, with named files, testable logic, and full visibility
  • Plain JS → lets you build real behavior without magic
  • HTML → already is the perfect component: bundled, visible, portable
  • A loop → is all the reactivity you need. One heartbeat driving your system. Fully testable. Fully controlled.
